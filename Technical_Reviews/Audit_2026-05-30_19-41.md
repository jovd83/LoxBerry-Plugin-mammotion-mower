## Executive Summary
- **Maintainability Score: 87 / 100** (start 100; −6 for 3 Medium, −7 for 7 Low).
- **State Of The Union:** This is a mature, carefully-engineered plugin, not a prototype. It shows real operational thought — `atexit` pidfile reclamation, password redaction in the login loop, permanent-vs-transient auth classification to avoid account lockouts, change-suppressed retained MQTT publishes, and a graceful soft-exit model. The defects that remain are **release-hygiene and consistency** issues, not architectural failures: a version/identity drift between the docs and the shipped metadata, one credential-redaction path that the safer login path already solved but a second path missed, a small amount of dead config/code, and no automated validation gate.
- **Top Risks:**
  1. **Version & documentation drift** — the repo ships `1.0.0` (plugin.cfg/release.cfg/CGI) but the README, its version badge, and the install instructions still say `0.1.2` and tell users to download `mammotion-mower-0.1.2.zip`; the CHANGELOG never records the 0.1.1/0.1.2 work and its only `1.0.0` entry is "author set to jovd83". A user following the README downloads a non-existent/old artifact.
  2. **Inconsistent credential redaction** — the login retry loop scrubs the password out of error text, but `_on_unrecoverable_auth` logs and **publishes the raw exception to the retained MQTT topic `_last_error`** without the same scrubbing. If PyMammotion ever echoes the submitted password in that exception, it persists in the broker and the daemon log.
  3. **No validation gate** — there is no `.github/workflows/`, no tests, and no lint config. For a plugin that bridges cloud credentials to a home-automation bus, the absence of even a syntax/JSON CI check raises regression risk on every release (and deviates from the user's own LoxBerry-plugin repo template standard).

## Architecture Snapshot
- **Entry Points:**
  - Boot / CGI: `daemon/daemon` (`start|stop|restart|status`) → forks `bin/mammotion-mower.py` as `loxberry`.
  - Web UI: `webfrontend/htmlauth/index.cgi` (renders `templates/settings.html`, writes `config/default.json`, calls the daemon wrapper `restart`).
  - Installer: `postinstall.sh` (Python 3.13 standalone bootstrap + venv + `pip install`).
- **Core Layers:**
  - *Presentation:* `index.cgi` + `settings.html` (HTML::Template).
  - *Config:* `config/default.json` (single source of truth; plaintext password mode 0600).
  - *Domain/Bridge:* `bin/mammotion-mower.py` — cloud login, continuous report stream, `extract_datapoints()` (mower struct → flat MQTT keys), `COMMAND_MAP` (MQTT → pymammotion).
  - *Infra:* paho MQTT publisher with LoxBerry broker auto-discovery from `general.json`; `mqtt_subscriptions.cfg` auto-registration for the MQTT Gateway.
- **Hot Files / Hot Paths:**
  1. `bin/mammotion-mower.py:497 amain()` — orchestrates the entire lifecycle (login → stream → command-subscribe → poll loop → shutdown). ~240 lines; the single most complex unit.
  2. `bin/mammotion-mower.py:308 extract_datapoints()` — the data-shape mapping that is most likely to drift against PyMammotion upstream.
  3. `webfrontend/htmlauth/index.cgi:27-95` — the POST/save path that persists credentials and restarts the daemon.
- **Constraints / Unknowns:**
  - PyMammotion is an unofficial, fast-moving reverse-engineered client; `extract_datapoints()` and `COMMAND_MAP` are coupled to its internal attribute names (`report_data.dev.battery_val`, `send_command_with_args`, etc.). These cannot be verified off a live device.
  - Perl/Python runtime behavior cannot be exercised on this Windows dev box.

## Refactor Sequence
Ordered by safety and value, behavior-preserving first:

1. **Security consistency (Medium #2):** add a single `redact_secret()` helper; route `_on_unrecoverable_auth` and all `_last_error`/error logs through it. *(Correctness/security, no behavior change for the happy path.)*
2. **Dead weight removal (Low):** drop `command_settle_seconds` (py + json) and unused `redact_url()`; hoist `import math` to module scope.
3. **CGI path correctness (Low):** derive the daemon path from `$lbhomedir` instead of the hardcoded `/opt/loxberry/...`.
4. **Docs/version truth (Medium #1):** align README (badge, install filename, body) to `1.0.0`; back-fill CHANGELOG 0.1.1/0.1.2/1.0.0 from git history.
5. **Delivery gate (Medium #3):** add `.github/workflows/validate.yml` (py_compile + `bash -n` + JSON + plugin.cfg sanity); optional pytest for `extract_datapoints`/`safe_segment`.
6. **Observability hardening (Low):** replace the prose-matched warning filter with a stable marker; add the blade-life `None`-resolution debug log (and verify the nested path).
